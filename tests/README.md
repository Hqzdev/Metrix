Tests

Этот документ описывает блок tests: структуру тестов проекта, правила запуска и правила добавления новых проверок.

Назначение

Блок tests хранит автоматические проверки для backend, Telegram-бота, интеграций и пользовательских сценариев.
Он не должен содержать production-код, seed-данные для приложения или ручные инструкции запуска сервисов.

В тестах можно использовать mock-объекты и in-memory реализации, если это проверяет конкретную бизнес-логику без внешних API.

Структура файлов

unit — быстрые unit-тесты без базы данных, Redis и внешних API
integration — интеграционные тесты репозиториев, API и очередей
e2e — пользовательские сценарии от входа до бронирования

unit/api — unit-тесты backend-блока packages/api
integration/.gitkeep — пустой маркер папки до появления интеграционных тестов
e2e/.gitkeep — пустой маркер папки до появления e2e-тестов

Запуск

Тесты запускаются из корня проекта:

```bash
npm test
```

Команда использует node:test и tsx.
Отдельный test runner не нужен.

Unit tests

unit-тесты должны проверять чистую логику:

* validators
* mappers
* messages
* keyboards
* event bus
* расчёт дат и слотов

Unit-тест не должен подключаться к PostgreSQL, Redis, Telegram API, Google Calendar API или Microsoft Graph API.

Integration tests

integration-тесты нужны для проверки нескольких слоёв вместе:

* Prisma repository
* создание бронирования через БД
* BullMQ queue с Redis
* calendar adapters через mock HTTP
* auth/session flow

Такие тесты могут требовать DATABASE_URL или REDIS_URL.
Если env не задан, тест должен быть пропущен, а не падать.

E2E tests

e2e-тесты проверяют пользовательский сценарий целиком:

* выбор локации
* выбор ресурса
* выбор слота
* оплата или mock-оплата
* создание бронирования
* отмена или перенос

E2E-тесты должны быть максимально близки к реальному поведению, но без настоящих платежей и внешних календарей.

Правила

Название файла должно заканчиваться на .test.ts.

Тесты должны быть детерминированными.
Нельзя зависеть от текущего времени без фиксированной даты или mock.

Нельзя использовать реальные токены, реальные платежи и реальные внешние календарные аккаунты.

Расширение

Добавление unit-теста:

1. выбрать папку внутри tests/unit
2. создать файл с суффиксом .test.ts
3. импортировать test из node:test
4. импортировать assert из node:assert/strict
5. проверить один конкретный сценарий

Добавление integration-теста:

1. выбрать папку внутри tests/integration
2. проверить наличие нужного env
3. подготовить тестовые данные
4. выполнить сценарий
5. очистить созданные данные

Добавление e2e-теста:

1. описать пользовательский сценарий
2. заменить внешние сервисы mock-адаптерами
3. проверить финальное состояние бронирования
4. не использовать production-токены
