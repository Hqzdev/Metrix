# Кодстайл

Этот документ задаёт базовый стиль кода для репозитория на TypeScript, Node.js и React.

## Принципы

- Понятность важнее “умного” кода.
- Код пишется для сопровождения, а не для демонстрации.
- Один модуль — одна зона ответственности.
- Явное поведение лучше скрытой магии.
- Сначала корректность и читаемость, потом оптимизация.

## Общие правила

- Весь новый код пишется на TypeScript.
- Обязательны комментарии.
- По умолчанию используем ES Modules.
- Используем `const`, если переменная не переопределяется.
- `let` используем только при реальном изменении значения.
- `var` не используем.
- Предпочитаем ранние `return`, а не глубокую вложенность.
- Не держим мёртвый код, закомментированные блоки и неиспользуемые импорты.
- Не оставляем `TODO` без контекста и понятного следующего шага.
- Если файл начинает решать несколько задач, его нужно делить.

## Именование

- `camelCase` для переменных, функций и полей объектов.
- `PascalCase` для компонентов, типов, интерфейсов, классов.
- `UPPER_SNAKE_CASE` только для настоящих констант.
- Избегаем неочевидных сокращений.
- Имя должно выражать смысл, а не техническую реализацию.

Хорошо:

```ts
const d = 500
function calc() {}
type data = {}
```

Плохо:

```ts
const retryDelayMs = 500
function calculateInvoiceTotal() {}
type UserProfile = {}
```

## TypeScript

- Для публичных API предпочитаем явные типы.
- Для локальных переменных используем вывод типа, если он очевиден.
- `type` используем по умолчанию.
- `interface` используем только там, где действительно нужна расширяемость.
- `any` запрещён, кроме очень редких интеграционных случаев.
- Если тип неизвестен, используем `unknown` и затем сужаем.
- Для сложных состояний предпочитаем discriminated unions.
- Явно отражаем `null` и `undefined` в типах.
- Если объект не должен меняться, это должно быть видно из кода.

Хорошо:

```ts
type ApiResult =
  | { status: "ok"; data: User }
  | { status: "error"; message: string }
```

Плохо:

```ts
type ApiResult = {
  ok: boolean
  data?: User
  error?: string
}
```

## Функции

- Функция должна делать одну вещь.
- По возможности функция должна быть чистой.
- Зависимости лучше передавать явно, а не тянуть из глобального контекста.
- Если параметров становится слишком много, объединяем их в объект.
- Не используем boolean-флаги, если они заметно меняют смысл функции.

Хорошо:

```ts
function createReport(options: { userId: string; includeDrafts: boolean }) {}
```

Плохо:

```ts
function createReport(userId: string, includeDrafts: boolean, isAdmin: boolean, fast: boolean) {}
```

## Ошибки

- Нельзя проглатывать ошибки молча.
- Сообщения об ошибках должны быть полезными.
- Внешний ввод валидируется на границах системы.
- Для ожидаемых ошибок лучше использовать явный результат, а не бросать исключение.
- Исключения используем для неожиданных или нештатных ситуаций.

## Node.js

- HTTP-слой, бизнес-логика и доступ к данным должны быть разделены.
- Нельзя смешивать route handler и доменную логику в одном месте.
- Доступ к `process.env` должен быть централизован.
- Переменные окружения валидируются при старте.
- Используем `async/await`, а не цепочки `.then`.
- Избегаем тяжёлых синхронных операций в request path.
- Логируем структурированно и по делу.

Рекомендуемая структура:

- `src/routes` — HTTP handlers
- `src/services` — бизнес-логика
- `src/repositories` — доступ к БД и внешним источникам
- `src/lib` — утилиты общего назначения
- `src/config` — runtime/env-конфигурация

## React

- Используем функциональные компоненты.
- Компонент должен быть сфокусирован на одной задаче.
- Сложную логику выносим в хуки или сервисы.
- UI-компоненты должны оставаться в основном декларативными.
- Важные формы делаем контролируемыми.
- Не создаём новые компоненты внутри `render`, если это не оправдано.
- Избегаем лишнего prop drilling, если композиция или context лучше.
- Эффекты должны быть минимальными и понятными.

## Структура React-компонента

Рекомендуемый порядок:

1. Импорты
2. Типы
3. Константы
4. Компонент
5. Производные значения
6. Обработчики
7. Эффекты
8. JSX

## Состояние

- Храним в state только то, что нельзя вычислить.
- Не дублируем производные значения в state.
- `useMemo` и `useCallback` используем только при реальной пользе.
- Для сложных переходов состояния лучше `useReducer`.
- Избегаем глубоко вложенного изменяемого состояния.

## Хуки

- Кастомные хуки начинаются с `use`.
- Один хук — одно повторно используемое поведение.
- Не прячем сайд-эффекты в обычные utility-функции.
- Dependency arrays должны быть корректными.
- Не отключаем lint по хукам без сильной причины.

## Комментарии

- Комментарии пишем только там, где кодом нельзя выразить смысл достаточно ясно.
- Объясняем “зачем”, а не “что”.
- Устаревшие комментарии удаляем сразу.

Хорошо:

```ts
// Ограничиваем число повторов, чтобы не спровоцировать дублирующий платёж.
```

Плохо:

```ts
// Увеличиваем i на 1
i++
```

## Импорты

- Группируем импорты:
  1. внешние пакеты
  2. внутренние абсолютные импорты
  3. относительные импорты
- Не оставляем неиспользуемые импорты.
- Если в проекте есть alias, используем его вместо длинных `../../../`.

## Стили и UI

- В одной зоне проекта используем один подход к стилям.
- Повторяющиеся цвета, размеры и отступы выносим в токены или общий слой.
- Избегаем магических чисел без причины.
- Если список классов начинает мешать чтению, выносим его в helper или переменную.

## Тестовое мышление

- Код проектируется так, чтобы его можно было тестировать.
- Побочные эффекты должны быть отделены от бизнес-правил.
- Бизнес-логику тестируем отдельно от транспорта и UI.
- Покрываем критические сценарии, пограничные случаи и ошибки.

## Форматирование

- Форматирование автоматическое и единообразное.
- Отступ — 2 пробела.
- Длина строки — разумная, без искусственного уплотнения.
- Политика по точкам с запятой должна быть единой во всём проекте.
- Трейлинг-коммы — по настройке форматтера.

## Имена файлов

- Для обычных файлов предпочитаем `kebab-case`.
- Для React-компонентов допустим `PascalCase`, если это принято в проекте.
- Имя файла должно соответствовать основному экспорту.

## Git-гигиена

- Один коммит — одна логическая задача.
- Не смешиваем большой рефакторинг и изменение поведения без необходимости.
- Переименования и переносы лучше делать отдельным коммитом.
- Перед merge код должен проходить форматирование, линт и типовую проверку.

## Базовый стандарт для этого репозитория

- Только TypeScript
- Только функциональные React-компоненты
- Без `any`
- Ранние `return`
- Бизнес-логика вне UI
- Валидация внешнего ввода на границах
- Маленькие модули вместо больших файлов
- Читаемость важнее лишней абстракции
