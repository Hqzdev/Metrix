# Архитектура Smart Booking System

Этот документ фиксирует целевую архитектуру системы бронирования переговорных комнат и рабочих мест. Архитектура выбрана с расчётом на быстрый запуск продукта, дальнейшее развитие интеграций и предсказуемую поддержку.

## Архитектурный подход

Система реализована как **микросервисная архитектура**. Каждый домен выделен в отдельный сервис с независимым деплоем, собственной зоной ответственности и изолированным доступом к данным.

Сервисы взаимодействуют двумя способами:

- **Синхронно** — HTTP-запросы с HMAC-подписью для service-to-service аутентификации и защиты от replay-атак.
- **Асинхронно** — Redis Streams для событий (`booking.created`, `booking.cancelled`) между сервисами.

Единственная публично доступная точка входа — `bot-gateway`. Все остальные сервисы работают внутри Docker-сети и не выставлены наружу.

### Сервисы платформы

| Сервис | Порт | Зона ответственности |
|---|---|---|
| `bot-gateway` | 3000 | Telegram polling, маршрутизация команд, rate limiting |
| `booking-service` | 3001 | Бронирования, слоты, локации, ресурсы |
| `calendar-service` | 3002 | Google Calendar OAuth, синхронизация событий |
| `payment-service` | 3003 | Telegram Payments / ЮКасса, pre-checkout |
| `notification-service` | — | Отправка Telegram-уведомлений по Redis Streams |
| `analytics-service` | 3005 | Агрегация метрик, отчёты |
| `admin-service` | 3006 | Административные операции |

### Разделяемые пакеты

Общий код вынесен в монорепозиторные пакеты внутри `apps/bot/packages/`:

- `@metrix/contracts` — типы и константы потоков Redis Streams
- `@metrix/auth` — HMAC-подпись запросов, проверка подписи, audit log
- `@metrix/redis-bus` — обёртка над Redis для pub/sub и replay protection

## Общая схема

Система состоит из следующих частей:

1. Web-приложение для сотрудников и администраторов.
2. Backend API для бронирований, доступности, пользователей, аналитики и интеграций.
3. Telegram-бот для быстрого бронирования и напоминаний.
4. Очереди фоновых задач для синхронизации календарей, уведомлений и генерации отчётов.
5. Realtime-канал для обновления доступности и статусов без ручного обновления страницы.

## Технологические решения

### Frontend

- `Next.js`

- `Tailwind CSS`
- `React Query` для серверного состояния
- `Redux Toolkit` только для действительно общего клиентского UI-состояния, а не для всех данных подряд

### Backend

- `Node.js`
- `TypeScript`
- `PostgreSQL`
- `Prisma ORM`

### Инфраструктура и асинхронность

- `Redis` для кэша, блокировок и служебных сценариев
- `BullMQ` для очередей фоновых задач
- `WebSocket` или `SSE` для realtime-обновлений

### Внешние интеграции

- `Google Calendar API`
- `Microsoft Graph API` для Outlook / Microsoft 365
- `Telegram Bot API`

## Главный архитектурный принцип

Слои и модули должны быть разделены так, чтобы:

- UI не содержал доменные правила;
- HTTP-слой не решал бизнес-задачи напрямую;
- интеграции не протекали в доменную модель;
- доменная логика не зависела от конкретного транспорта и UI.

## Доменная модель

### Основные сущности

- `User` — сотрудник или администратор
- `Room` — переговорная комната
- `Desk` — рабочее место / место в коворкинге
- `Booking` — бронь помещения или рабочего места
- `BookingSlot` — интервал доступности
- `CalendarConnection` — подключение внешнего календаря пользователя или ресурса
- `Notification` — напоминание или служебное уведомление
- `UsageMetric` — агрегированные показатели использования
- `Report` — экспортный аналитический отчёт

### Ключевые инварианты

- один и тот же ресурс нельзя забронировать дважды на пересекающееся время;
- отмена брони должна отменять связанное событие во внешнем календаре;
- перенос брони должен быть атомарным сценарием;
- внешняя синхронизация не должна ломать внутреннюю консистентность;
- аналитика строится на подтверждённых фактах бронирования, а не на предположениях UI.

## Модули системы

Рекомендуемые доменные модули:

- `auth` — аутентификация, авторизация, роли, сессии

- `resources` — переговорные, рабочие места, правила доступности

- `telegram` — bot flow, команды, inline-кнопки, уведомления
- `notifications` — напоминания и сервисные уведомления
- `analytics` — агрегирование и визуализация метрик
- `reports` — экспорт PDF и выгрузки
- `admin` — административные настройки и мониторинг

## Внутренние слои

Внутри каждого доменного модуля желательно придерживаться одинаковой структуры:

```txt
module-name/
  domain/
  application/
  infrastructure/
  presentation/
```

### `domain`

Содержит:

- сущности;
- value objects;
- доменные правила;
- доменные события;
- интерфейсы репозиториев и портов.

### `application`

Содержит:

- use cases;
- orchestration сценариев;
- DTO команд и результатов;
- координацию нескольких доменных объектов.

### `infrastructure`

Содержит:

- реализацию репозиториев через Prisma;
- адаптеры к Redis, BullMQ и внешним API;
- мапперы между внешними контрактами и доменными моделями.

### `presentation`

Содержит:

- HTTP handlers;
- схемы валидации;
- сериализацию ответов;
- websocket events;
- интерфейс для Telegram-сценариев, если он относится к модулю.

## Межсервисное взаимодействие

### Синхронная коммуникация (HTTP)

Сервис-вызывающий подписывает запрос HMAC-SHA256 по схеме: метод + путь + тело + timestamp. Вызываемый сервис проверяет подпись и timestamp (окно ±30 сек). Каждый запрос содержит уникальный `X-Request-Id`, который проверяется через Redis на повторное использование — это защита от replay-атак.

Каждый сервис явно перечисляет доверенных вызывающих через env-переменные `TRUSTED_*_SECRET`. Если секрет не задан — сервис отклоняет все запросы.

### Асинхронная коммуникация (Redis Streams)

События публикуются в именованные стримы из `@metrix/contracts`:

- `STREAMS.BOOKING_CREATED` — создание брони
- `STREAMS.BOOKING_CANCELLED` — отмена брони

Подписчики (`notification-service`, `analytics-service`, `calendar-service`) читают события независимо и не блокируют основной flow бронирования.

### Идентификация пользователя

Telegram user ID передаётся между сервисами через заголовок `X-User-Id`, подписанный отдельным секретом `USER_ID_SIGNING_SECRET`. Это исключает подделку идентификатора на уровне тела запроса.

## Работа с бронированиями

### Сценарий создания брони

1. Пользователь выбирает ресурс и временной слот.
2. Система валидирует права доступа и правила использования ресурса.
3. Система проверяет пересечения по времени.
4. Бронь создаётся в транзакции.
5. Публикуется событие `booking.created`.
6. Фоновая задача создаёт событие во внешнем календаре.
7. Realtime-канал обновляет доступность у подключённых клиентов.

### Защита от двойного бронирования

Нужно использовать сразу несколько уровней защиты:

- проверка пересечений на уровне application-логики;
- транзакция в БД;
- ограничение или уникальный механизм на уровне схемы данных;
- при высоком уровне конкуренции можно добавить короткую Redis-блокировку на ресурс и временной диапазон.

## Realtime

Realtime нужен не для всех экранов, а для следующих сценариев:

- список свободных переговорных;
- календарная сетка доступности;
- изменение статуса брони;
- панель администратора с оперативным мониторингом.

Если нужен только односторонний поток обновлений, предпочтительнее `SSE`. Если нужны двусторонние события и интерактивные клиентские сценарии, используем `WebSocket`.

## Очереди и фоновые задачи

`BullMQ` используется только для операций, которые не должны блокировать HTTP-ответ:

- создание события во внешнем календаре;
- удаление события при отмене;
- повторные попытки синхронизации после временных ошибок;
- отправка напоминаний в Telegram;
- генерация PDF-отчётов;
- пересчёт аналитических агрегатов.

Очередь не должна скрывать критическую бизнес-логику. Основной факт бронирования фиксируется синхронно в БД.

## Интеграции и Adapter Pattern

Интеграции нужно изолировать через адаптеры. Это особенно важно, потому что:

- Google и Microsoft имеют разные модели событий и авторизации;
- Telegram имеет отдельный transport и отдельные ограничения по взаимодействию;
- бизнес-логика не должна зависеть от формата ответа внешнего API.

Пример подхода:

- `CalendarProvider` — общий контракт
- `GoogleCalendarAdapter` — реализация для Google
- `MicrosoftCalendarAdapter` — реализация для Microsoft

То же правило применяется к Telegram-командам и уведомлениям.

## Хранение данных

### PostgreSQL

Основное хранилище:

- пользователи;
- помещения и рабочие места;
- бронирования;
- связи с внешними календарями;
- история действий;
- агрегированные аналитические таблицы.

### Redis

Используется как вспомогательный слой:

- кэш доступных слотов;
- блокировки при конкурентных бронированиях;
- storage для очередей;
- rate limiting;
- временные ключи для служебных сценариев.

## Безопасность

Обязательные требования:

- только `HTTPS`;
- `OAuth2` для Google и Microsoft;
- безопасное хранение refresh/access tokens интеграций;
- сессии через `JWT` или серверную session-стратегию с короткоживущими токенами;
- хеширование паролей через `argon2` или `bcrypt`, если есть локальная авторизация;
- проверка ролей и прав на уровне use case, а не только в UI;
- аудит критических действий администратора.

## Аналитика

Аналитика не должна считаться тяжёлыми запросами в реальном времени при каждом открытии страницы.

Подход:

- ключевые события пишутся в таблицы фактов;
- агрегаты пересчитываются по расписанию или по событию;
- UI читает готовые или почти готовые агрегированные данные.

Основные метрики:

- карта занятости по часам и дням;
- процент использования каждого ресурса;
- пиковые часы бронирований;
- частота отмен и переносов;
- активность по подразделениям или локациям, если это появится в продукте.

## API-контракты

Контракты должны быть разделены по модулям:

- `/auth`
- `/resources`
- `/bookings`
- `/availability`
- `/integrations`
- `/analytics`
- `/reports`
- `/admin`

Правила:

- внешние DTO не должны совпадать с Prisma-моделями один в один;
- вход валидируется на границе;
- ответы должны быть стабильными и версионируемыми;
- ошибки должны быть нормализованы и понятны клиенту.

## Нефункциональные требования и архитектурные последствия

### Производительность

Если целевой SLA:

- API `< 500 мс` для 95% запросов;
- загрузка страницы `< 3 сек`;

то архитектурно это означает:

- минимальные запросы в critical path;
- кэширование справочников и доступности там, где это оправдано;
- тяжёлые операции выносить в очередь;
- аналитические данные не считать на лету из сырых таблиц.

### Доступность

Целевой `99% uptime` означает:

- health checks;
- retry policy для внешних API;
- idempotency для задач синхронизации;
- мониторинг ошибок очередей и интеграций.

## Масштабирование сервисов

Каждый сервис можно масштабировать независимо. Наиболее вероятные кандидаты при росте нагрузки:

- `booking-service` — центральный сервис, через него проходит весь критический путь
- `calendar-service` — синхронизация с Google Calendar может быть медленной при большом числе пользователей
- `analytics-service` — агрегация исторических данных может стать дорогой операцией

`notification-service` масштабируется естественно: он только читает из Redis Streams и отправляет сообщения в Telegram.

## Архитектурные запреты

- не смешивать UI и доменные правила;
- не делать `bot-gateway` источником истины по бронированиям — он только маршрутизирует запросы;
- не обращаться к `booking-service` напрямую в обход HMAC-подписи;
- не хранить внешние API-ответы как доменную модель без маппинга;
- не строить аналитику прямыми тяжёлыми запросами по каждому открытию страницы;
- не добавлять прямые HTTP-связи между сервисами, минуя `@metrix/auth` — все межсервисные запросы должны быть подписаны;
- не использовать Redux как замену backend state management.
