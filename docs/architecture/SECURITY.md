# Security

Этот документ объясняет безопасность Metrix простым языком.

## Главная идея

Система не должна доверять запросу просто потому, что он пришёл по сети.
Каждый важный запрос должен быть проверен — и внешний, и внутренний.
Один скомпрометированный компонент не должен открывать доступ ко всему остальному.

## Что защищаем

- брони и слоты
- платежи и инвойсы
- Telegram user id
- refresh tokens сессий
- refresh tokens календарей
- admin-действия
- внутренние сервисы
- audit log

---

## HMAC для внутренних сервисов

Когда один сервис вызывает другой, он подписывает запрос своим секретом.

В подпись входят:

- HTTP method
- path
- timestamp (защита от replay без Redis)
- request id (replay protection через Redis)
- SHA-256 хеш тела запроса

Если подпись неправильная — запрос отклоняется. Если подпись старая (clock skew > 30 секунд) — тоже.

Файл: `apps/bot/packages/auth/src/index.ts`

---

## Replay protection

Даже правильный запрос нельзя выполнить дважды.

Сервис запоминает `X-Request-Id` в Redis на 60 секунд.
Если такой id приходит снова — это replay, запрос отклоняется.

Файл: `apps/bot/packages/redis-bus/src/index.ts` → `checkReplay`

## JWT с версионированием ключа (kid)

Access token — короткоживущий JWT (15 минут).

В header JWT записывается `kid` — идентификатор ключа.
При проверке токена система смотрит `kid` и выбирает нужный ключ из набора.

Это позволяет ротировать секрет без мгновенного разлогина всех пользователей:
1. Добавить новый ключ в `current`, старый переместить в `previous`.
2. Новые токены подписываются новым ключом.
3. Старые токены продолжают работать до истечения (15 минут).
4. После 15 минут убрать старый ключ из `previous`.

Файл: `packages/api/src/shared/auth/jwt.ts`

---

## Одноразовые refresh токены

Refresh token — длинная случайная строка (32 байта), хранится в базе данных.

Каждый refresh token используется **один раз**.
При обновлении сессии:
1. Старый refresh token удаляется из базы.
2. Создаётся новый refresh token.
3. Оба шага выполняются атомарно (в одной транзакции).

Если тот же refresh token придёт повторно — его уже не будет в базе, и запрос получит ошибку.
Это означает либо попытку replay, либо утечку токена.

Файл: `packages/api/src/shared/auth/session-service.ts` → `rotateSession`

---

## Blacklist отозванных токенов

Access token живёт 15 минут. Если пользователь разлогинился или сменил пароль,
ждать 15 минут небезопасно — токен всё это время остаётся валидным.

При logout или смене пароля:
1. Access token добавляется в Redis blacklist (хранится SHA-256 хеш, не сам токен).
2. При каждой аутентификации система проверяет blacklist.
3. Запись в blacklist удаляется сама через BLACKLIST_TTL_SECONDS (16 минут).

Файл: `packages/api/src/shared/auth/token-blacklist.ts`

---

## Brute-force защита на вход

Система считает неудачные попытки входа отдельно по IP и по userId.

Правила:
- до 5 неудач — вход разрешён
- 5 неудач → блокировка на 1 минуту
- каждая следующая неудача удваивает блокировку (2 мин, 4 мин, 8 мин...)
- максимум 30 минут
- при успешном входе счётчик сбрасывается

Счётчики хранятся в Redis и удаляются автоматически.

Файл: `packages/api/src/shared/auth/login-rate-limiter.ts`

## Signed user identity

bot-gateway подписывает Telegram user id перед передачей во внутренние сервисы.
Внутренний сервис проверяет подпись через HMAC.

Это нужно, чтобы никто не мог передать произвольный user id в теле запроса.

Файл: `apps/bot/packages/auth/src/index.ts` → `signUserId` / `extractUserId`

---

## OAuth state

OAuth state подписывается перед redirect в Google.
При возврате подпись проверяется — это защищает от CSRF и подмены данных.

Файл: `apps/bot/packages/auth/src/index.ts` → `signOAuthState` / `verifyOAuthState`

---

## Token encryption

Refresh tokens Google Calendar хранятся зашифрованными в базе данных.
Если база утечёт — токены без ключа шифрования бесполезны.

---

## Rate limit (Telegram bot)

bot-gateway ограничивает частоту действий пользователя через Redis.

Текущее правило: 10 запросов за 10 секунд на одного пользователя.
Счётчик хранится в Redis и работает корректно при нескольких экземплярах bot-gateway.

Файл: `apps/bot/services/bot-gateway/src/rate-limiter.ts`

---

## RBAC

Admin-команды доступны только администраторам.
Обычный пользователь не должен попасть в admin flow.

Все отказы в доступе логируются с причиной.

Файл: `apps/bot/packages/rbac/src/index.ts`

---

## Audit log

Важные действия пишутся в audit log:

- создание брони
- отмена брони
- payment recovery
- DLQ replay
- admin updates
- запрещённые действия (RBAC denied)
- logout и отзыв токенов

Audit log пишется в stdout в структурированном JSON и в таблицу `AuditLog` в базе.

---

## CSP (Content Security Policy)

Веб-приложение использует строгий CSP с nonce.

Для каждого запроса генерируется случайный nonce.
Только скрипты с этим nonce выполняются браузером.

Что убрано:
- `unsafe-eval` — разрешал `eval()` и `Function()`, основной XSS вектор
- `unsafe-inline` для скриптов — заменён на nonce

Файл: `apps/web/proxy.ts`

---

## Docker boundary

PostgreSQL, Redis и внутренние сервисы не публикуются наружу.
Наружу открыт только bot-gateway health port и веб-приложение.

## Dependency security

`npm audit --audit-level=high` запускается автоматически перед каждым push.
Если есть критические уязвимости в зависимостях — push блокируется.

## Что делать при изменениях

**Новый внутренний endpoint:**
1. Добавь HMAC-проверку через `verifyServiceRequest`.
2. Добавь replay protection через `checkReplay`.
3. Проверь user identity, если endpoint действует от имени пользователя.
4. Запиши audit log для опасных действий.

**Ротация JWT секрета:**
1. Добавь новый ключ как `current`, старый переместить в `previous`.
2. Подожди 15 минут (TTL access token).
3. Убери старый ключ из `previous`.

**Компрометация refresh token:**
1. Удали все сессии пользователя из базы (`Session` table).
2. Попроси пользователя войти заново.

**Компрометация access token:**
1. Вызови `revokeAccessToken(token, redis)` — токен немедленно попадёт в blacklist.
2. Удали сессию пользователя из базы.
