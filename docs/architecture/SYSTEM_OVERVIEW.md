System Overview

Этот документ описывает систему на верхнем уровне: части продукта, связи между ними, границы web, backend, бота, интеграций и аналитики.

Назначение

Документ фиксирует общую форму системы.
Он не описывает отдельные файлы кода.
Он нужен, чтобы разработчик сначала понял систему целиком, а уже потом переходил к модулям, API и БД.

Что описывает документ

контекст системы
акторов: сотрудник, администратор, внешние сервисы
основные потоки данных
high-level схему компонентов
границы ответственности между подсистемами
нефункциональные ограничения, которые влияют на архитектуру

Основные части системы

web — пользовательский и административный интерфейс
backend — HTTP API и application-слой
bot — Telegram-интерфейс и сценарии бронирования
integrations — внешние API и адаптеры
analytics — чтение фактов и расчёт агрегатов
queues — фоновые задачи и асинхронные процессы
shared — общий код, конфигурация, ошибки, логирование

Границы ответственности

web не содержит бизнес-логику.
web обращается к backend через API.

backend выполняет use cases, проверяет права и управляет доменными операциями.

bot не работает напрямую с хранилищем.
bot вызывает сервисы и формирует ответы для Telegram.

integrations изолируют внешние API.
Доменная логика не должна зависеть от формата Google, Microsoft или Telegram.

analytics читает данные и строит агрегаты.
analytics не создаёт бронирования и не изменяет ресурсы.

queues выполняют долгие операции вне синхронного запроса.

Архитектурное решение

На старте используется модульный монолит.
Это снижает стоимость разработки и сохраняет явные границы между модулями.
Микросервисы не вводятся до появления реальной потребности в независимом масштабировании, деплое или ownership.
