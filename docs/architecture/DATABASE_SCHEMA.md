Database Schema

Этот документ описывает схему данных проекта: основные сущности, связи, ограничения и технические правила хранения данных.

Назначение

Документ нужен, чтобы схема БД не проектировалась на лету по мере написания контроллеров и UI.
Сущности, связи и ограничения должны быть согласованы до реализации бизнес-сценариев.

AuditLog

AuditLog хранится в schema audit.

Поля:

id — внутренний идентификатор записи
ts — время события
service — сервис-источник
action — бизнес-действие
actorUserId — Telegram user id, если действие связано с пользователем
entityType — тип сущности
entityId — id сущности
requestId — request id межсервисного запроса
callerService — вызывающий сервис
payload — дополнительные безопасные детали события

Индексы:

service + action + ts — поиск событий по типу
entityType + entityId — история конкретной сущности

PendingInvoice

PendingInvoice хранит платежный факт и не удаляется после оплаты.

Статусы:

pending — invoice ждёт оплаты
paid_part — часть оплаты принята, создан следующий invoice
completed — invoice полностью оплачен
failed — invoice связан с failed saga или компенсацией
expired — hold истёк до завершения оплаты

Поля:

completedAt — время завершения оплаты
supersededByInvoiceId — следующий invoice при многочастной оплате

Правила проектирования

Бронирования должны иметь защиту от пересечений.
Оплата должна создавать временный hold до финального Booking.
Активный hold должен защищать слот от второго invoice.
Внешние календарные события должны быть связаны с внутренней бронью.
Токены календарей не должны храниться открытым текстом.
Аудит должен фиксировать мутирующие действия.
Audit log должен храниться в PostgreSQL, stdout logs не считаются единственным источником аудита.
Аналитические агрегаты не должны заменять исходные факты бронирований.
