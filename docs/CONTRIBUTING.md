# Правила внесения изменений

## Общий подход

- Пишите код так, как будто его будет поддерживать другой человек через полгода.
- Перед изменением сначала поймите контекст, потом редактируйте.
- Если изменение затрагивает архитектуру, зафиксируйте это в документации.

## Перед началом работы

1. Прочитайте `codestyle.md`.
2. Прочитайте `ARCHITECTURE.md`.
3. Уточните задачу и границы изменения.
4. Не делайте побочных рефакторингов без необходимости.

## Как вносить изменения

- Один pull request — одна логическая задача.
- Не смешивайте фиксы, рефакторинг и форматирование в одном изменении без причины.
- Если меняется контракт, обновите типы и документацию.
- Если добавляется новая feature, держите код внутри её зоны ответственности.

## Требования к коду

- Код должен соответствовать `codestyle.md`.
- Не добавляйте `any`, если без этого можно обойтись.
- Не оставляйте дебажные `console.log`.
- Не коммитьте временные файлы и локальные артефакты.

## Требования к структуре

- UI-компоненты не должны тащить в себя бизнес-логику.
- Route handlers должны быть тонкими.
- Работа с БД и внешними API выносится в repositories/adapters.
- Общие утилиты не должны становиться мусорным контейнером.

## Коммиты

Рекомендации:

- Коммит должен быть сфокусированным.
- Сообщение коммита должно быть коротким и понятным.

Хорошо:

- `feat: add metrics aggregation service`
- `fix: handle null API payload in dashboard`
- `refactor: extract auth session parser`

Плохо:

- `update`
- `fix stuff`
- `changes`

## Pull Request

PR должен отвечать на три вопроса:

1. Что изменено?
2. Зачем это сделано?
3. Как это проверялось?

### В описание PR включать

- краткую цель
- список ключевых изменений
- риски или ограничения
- шаги проверки

## Проверка перед merge

Минимум:

- код отформатирован
- линт проходит
- типовая проверка проходит
- критические сценарии вручную проверены

Если есть тесты:

- изменённая логика покрыта тестами

## Документация

Документацию нужно обновлять, если вы:

- меняете архитектурное решение
- добавляете новый слой или новую feature
- вводите новые соглашения
- меняете workflow разработки

## Чего делать не нужно

- не протаскивать крупный рефакторинг под видом мелкого фикса
- не смешивать стильовые правки с логикой без причины
- не копировать код, если можно выделить повторяемый слой
- не скрывать сложность за абстракцией, если она не окупается

## Если решения неочевидны

- оставьте короткое объяснение в PR
- при необходимости добавьте комментарий в код с объяснением причины
- лучше простое и прозрачное решение, чем “гибкое”, но трудно сопровождаемое
