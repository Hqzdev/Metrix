Decision

Использовать HMAC-SHA256 для service-to-service authentication между внутренними HTTP-сервисами apps/bot.

Context

Внутренние сервисы Metrix не должны доверять запросу только потому, что он пришёл из Docker network.
Компрометация одного сервиса не должна автоматически давать доступ ко всем операциям системы.

Нужна схема, которая проверяет:

имя вызывающего сервиса
целостность запроса
свежесть запроса
уникальность запроса

Options

Shared INTERNAL_SECRET:

простая настройка
легко ошибиться
компрометация одного секрета открывает всю систему
сложно ротировать по сервисам

JWT между сервисами:

стандартный формат
можно добавить claims
требует отдельной схемы выпуска и ротации токенов
для простых внутренних HTTP-вызовов добавляет лишний слой

HMAC per caller service:

подпись покрывает method, path, timestamp, request id и body hash
у каждого caller свой secret
принимающий сервис хранит список доверенных caller
схема простая и прозрачная для тестирования

Decision

Использовать HMAC-SHA256 с отдельными signing secrets для каждого вызывающего сервиса.
Replay protection выполнять через X-Request-Id и Redis TTL.

Consequences

Каждый внутренний HTTP-клиент должен подписывать запрос.
Каждый принимающий сервис должен явно описывать trusted callers.
Env-файлы становятся длиннее, но blast radius при компрометации ниже.
В тестах нужно проверять expired timestamp, invalid signature и повторный request id.

Status

Accepted.
