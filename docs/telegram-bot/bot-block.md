Telegram Bot src/bot

Этот документ описывает блок apps/bot/src/bot: обработку обновлений, распределение ответственности между файлами и правила добавления новой логики.

Назначение

Блок src/bot реализует слой взаимодействия с Telegram.
Он принимает обновления через TelegramClient, запрашивает данные у BookingService и формирует ответы (текст и inline-кнопки).

В этом слое не должна находиться бизнес-логика, работа с хранилищем, HTTP-взаимодействие с Telegram, работа с переменными окружения или генерация календарных ссылок.

Структура файлов

create-bot.ts — основной runtime: long polling, маршрутизация команд и пользовательский сценарий бронирования
admin-controller.ts — логика административной панели
payment-controller.ts — обработка платежей (инвойсы, pre-checkout, подтверждение оплаты)
keyboards.ts — генерация inline-клавиатур и callback payload
messages.ts — формирование текстов для пользователя

Runtime Flow

Функция createBot(options) создаёт экземпляр BotApp.
Метод start() регистрирует команды и запускает long polling.

Последовательность работы:

1. TelegramClient.getUpdates() получает обновления
2. сохраняется updateOffset для исключения повторной обработки
3. обновления маршрутизируются по типу:

   * message → handleMessage()
   * pre_checkout_query → PaymentController.handlePreCheckoutQuery()
   * callback_query → handleCallbackQuery()

Ошибки логируются и обрабатываются повторной попыткой.

Telegram допускает только один активный getUpdates-процесс на токен. Параллельный запуск приведёт к конфликту.

Команды

Обрабатываются в create-bot.ts:

/start — отображение главного меню
/help — список доступных команд
/admin — доступ к административной панели (при наличии прав)
/book — выбор локации
/slots — просмотр доступных слотов
/my_bookings — список активных бронирований

Если администратор находится в режиме редактирования, входящее сообщение интерпретируется как значение для изменения, а не как команда.

Callback routing

Inline-кнопки используют callback_data с префиксами.

admin:* — обработка в AdminController
menu:start — главное меню
menu:help — справка
menu:book, menu:slots — выбор локации
location:{locationId} — список ресурсов
resource:{locationId}:{resourceId} — список слотов
slot:{resourceId}:{slotId} — подтверждение
confirm:{resourceId}:{slotId} — запуск оплаты (PaymentController)
menu:bookings — список бронирований
cancel:{bookingId} — подтверждение отмены
cancel_confirm:{bookingId} — выполнение отмены

Размер callback_data ограничен. Используются только идентификаторы, остальные данные восстанавливаются через сервис.

Административная панель

Администрирование реализовано в AdminController.
Доступ определяется списком ADMIN_TELEGRAM_IDS.

Поддерживаемые операции:

* просмотр локаций
* изменение occupancy и количества участников
* просмотр ресурсов
* изменение цены, occupancy и статуса

Редактирование выполняется в два шага:

1. выбор действия через кнопку
2. ввод нового значения текстом

Состояние редактирования хранится в AdminEditState и очищается после успешного обновления.

Цена вводится как положительное число и преобразуется в:

* priceLabel
* priceMinorUnits

Платежи

PaymentController управляет процессом оплаты.
Один инвойс не должен быть больше 99 000 ₽.
Если ресурс стоит дороже, сумма разбивается на несколько платежей.

Подтверждение:

1. обработка confirm:{resourceId}:{slotId}
2. проверка актуальности данных
3. расчёт частей оплаты
4. создание invoice payload для первой части
5. отправка инвойса

Pre-checkout:

1. проверка payload
2. проверка пользователя, суммы и валюты
3. проверка доступности слота
4. проверка лимита 99 000 ₽ на одну часть
5. ответ Telegram

После оплаты:

1. сопоставление payload
2. подсчёт уже оплаченной суммы
3. если оплачены не все части — отправка следующего инвойса
4. если оплачены все части — создание бронирования через BookingService
5. отправка подтверждения
6. добавление ссылок на календарь
7. удаление payload

Payload хранится в памяти и в data/invoice-store.json.
Каждая часть оплаты получает отдельный invoice payload.

Пример:

* заказ на 214 000 ₽ разбивается на 99 000 ₽, 99 000 ₽ и 16 000 ₽
* бронирование создаётся только после третьего платежа

Сообщения и клавиатуры

messages.ts отвечает только за текст
keyboards.ts — только за кнопки

messages.ts не должен обращаться к сервисам или выполнять побочные действия.
Префиксы callback должны быть согласованы с обработчиками.

Зависимости

src/bot использует:

* BookingService
* TelegramClient
* типы Telegram
* генерацию календарных ссылок
* обработчики команд

Прямой доступ к данным, env или хранилищу запрещён. Все зависимости передаются через createBot.

Расширение

Добавление команды:

1. регистрация в setMyCommands
2. обработка в handleMessage
3. добавление сообщений и клавиатур при необходимости

Добавление callback-экрана:

1. выбор префикса
2. добавление кнопок
3. маршрутизация
4. получение данных через сервис

Добавление административного действия:

1. расширение AdminEditState
2. добавление кнопки
3. обработка callback
4. применение изменений через сервис

Добавление платёжной логики:

1. реализация в payment-controller.ts
2. источник цены — BookingResource.priceMinorUnits
3. обязательная валидация суммы и валюты перед подтверждением оплаты
