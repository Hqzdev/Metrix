# 5 ключевых правил кодстайла

---

## 1. Единый форматтер (обязательно)

**Что делаем:**
- Используем один форматтер (например, Prettier)
- Включаем автоформатирование на commit / CI

**Правило:**
- Код не форматируется вручную
- PR с неправильным форматированием не проходит

**Почему это важно:**
Форматирование не должно быть предметом обсуждения. Любая дискуссия на эту тему — потеря времени.

---

## 2. Строгий линтинг

**Что делаем:**
- Используем ESLint с жёсткими правилами
- Подключаем линтер к CI

**Правило:**
- Ошибки линтера блокируют merge
- `any`, мёртвый код и сомнительные конструкции запрещены
- Отключение правил требует явного обоснования

**Почему это важно:**
Без линтинга кодстайл разрушается не визуально, а логически — через баги и нестабильные паттерны.

---

## 3. Контроль размера и ответственности

**Что делаем:**
- Делим код на маленькие модули
- Следим за зоной ответственности

**Правило:**
- Одна функция — одна задача
- Один файл — одна зона ответственности
- Не смешиваем UI, бизнес-логику и доступ к данным

**Почему это важно:**
Основная проблема больших проектов — не стиль, а неконтролируемая сложность.

---

## 4. Единые контракты (типы и API)

**Что делаем:**
- Строго типизируем публичные интерфейсы
- Стандартизируем ответы API

**Правило:**
- Нет “неявных” структур данных
- Все входы и выходы описаны типами
- Ошибки и успешные ответы имеют единый формат

**Почему это важно:**
Без контрактов разработчики начинают “угадывать”, как работает система. Это главный источник багов.

---

## 5. Обязательный Code Review

**Что делаем:**
- Вводим обязательный review для всех изменений

**Правило:**
Ревью проверяет:
- читаемость
- простоту
- архитектурные границы
- соответствие стандартам

Ключевой критерий:
> Код должен быть понятен другому разработчику за ~30 секунд

**Почему это важно:**
Даже идеальные правила бесполезны, если их никто не соблюдает.

---

## Итог

Кодстайл в больших командах — это не про форматирование.

Это про:
- предсказуемость
- контроль сложности
- единое понимание кода

Если отсутствует хотя бы один из этих пунктов:
- форматтер
- линтер
- ограничения на структуру
- типы и контракты
- строгий review

→ кодовая база неизбежно деградирует.
