# Кодстайл

Этот документ задаёт стандарт кода для Metrix на TypeScript, Node.js и React.

Цель стандарта — код, который можно поддерживать командой годами: предсказуемый, скучный, безопасный и понятный без устных объяснений.

## Главный принцип

Код пишется для команды, а не для автора.

Код будут читать разработчики, QA, DevOps, security engineers и сам автор через несколько месяцев. Если поведение кода нужно объяснять голосом, значит границы, имена или структура выбраны плохо.

## Язык кода

- Код и имена сущностей пишутся на английском.
- Комментарии и JSDoc в этом проекте пишутся на русском.
- Документация продукта пишется на русском, если это удобно команде.
- Сообщения ошибок для внутренних API пишутся на английском.
- Пользовательские тексты пишутся на языке интерфейса.

## Общие правила

- Весь новый код пишется на TypeScript.
- Используем ES Modules.
- Используем `const`, если значение не переопределяется.
- `let` используем только при реальном изменении значения.
- `var` запрещён.
- Предпочитаем ранние `return`, а не глубокую вложенность.
- Не оставляем мёртвый код, закомментированные блоки и неиспользуемые импорты.
- Не оставляем `TODO` без контекста, owner или понятного следующего шага.
- Понятность важнее “умного” кода.
- Явное поведение лучше скрытой магии.

## Комментарии

Комментарии объясняют почему, а не что.

Плохо:

```ts
// увеличиваем счётчик.
count++
```

Хорошо:

```ts
// локальный счётчик снижает количество записей в Redis за текущее окно.
count++
```

Комментарии нужны для:

- бизнес-причин;
- нестандартных решений;
- security-sensitive логики;
- side effects;
- временных компромиссов с явным следующим шагом.

Устаревшие комментарии удаляются сразу.

## Публичная документация кода

Публичные функции, классы, сервисы, critical methods и API helpers документируются через JSDoc.

JSDoc должен описывать:

- назначение;
- важные side effects;
- security или consistency ограничения;
- условия ошибок, если они важны для caller.

Пример:

```ts
/**
 * обновляет ресурс через admin-границу.
 *
 * важно:
 * - в booking-service передаются только разрешённые поля.
 * - журнал аудита записывается после успешного обновления на стороне сервиса.
 * - запрос подписывается межсервисными учётными данными.
 */
async function updateResource() {}
```

## Именование

- `camelCase` для переменных, функций и полей объектов.
- `PascalCase` для компонентов, типов, интерфейсов и классов.
- `UPPER_SNAKE_CASE` только для настоящих констант.
- Не используем неочевидные сокращения.
- Имя должно выражать бизнес-смысл и side effects.

Плохо:

```ts
const d = 500
function calc() {}
type data = {}
```

Хорошо:

```ts
const RETRY_DELAY_MS = 500
function calculateInvoiceTotal() {}
type UserProfile = {}
```

## Никакой магии

Запрещены неименованные magic values в бизнес-логике.

Плохо:

```ts
if (status === 7) {}
```

Хорошо:

```ts
const STATUS_ARCHIVED = 7

if (status === STATUS_ARCHIVED) {}
```

Для доменных состояний предпочтительны string unions или enums.

## TypeScript

- Для публичных API обязательны явные типы.
- Для локальных переменных используем вывод типа, если он очевиден.
- `type` используем по умолчанию.
- `interface` используем только там, где нужна расширяемость.
- `any` запрещён, кроме редких интеграционных boundary случаев.
- Если тип неизвестен, используем `unknown` и затем сужаем.
- Для сложных состояний используем discriminated unions.
- `null` и `undefined` явно отражаются в типах.
- Не используем type assertions вместо validation на входе.

Хорошо:

```ts
type ApiResult =
  | { status: 'ok'; data: User }
  | { status: 'error'; message: string }
```

Плохо:

```ts
type ApiResult = {
  ok: boolean
  data?: User
  error?: string
}
```

## Структура проекта

Структура важнее красивого отдельного файла.

Крупные проекты ломаются от хаоса, cyclic dependencies, giant services и отсутствия boundaries.

Правила:

- один файл — одна ответственность;
- HTTP-слой не содержит бизнес-логику;
- validation находится на границе системы;
- repositories отвечают за данные;
- services/use cases отвечают за сценарии;
- adapters отвечают за внешние API;
- shared-код не должен становиться свалкой.

Рекомендуемая структура Node-сервиса:

```txt
src/config
src/routes
src/services
src/repositories
src/validators
src/middlewares
src/lib
```

Если файл требует постоянного scroll, его нужно делить.

## Функции

- Функция делает одну вещь.
- Ориентир длины — 20–40 строк.
- По возможности функция должна быть чистой.
- Зависимости передаются явно.
- Если параметров много, объединяем их в объект.
- Boolean-флаги запрещены, если они заметно меняют смысл функции.
- Side effects должны быть видны из имени.

Плохо:

```ts
processUser()
```

Хорошо:

```ts
saveUserAndSendWelcomeEmail()
```

## Validation

Нельзя доверять input.

Validation обязательна для:

- HTTP body;
- query params;
- path params;
- env;
- internal service requests;
- admin panel input;
- queue payloads;
- webhook payloads.

Frontend или internal service не считаются гарантией корректности.

Запрещены silent fallback для обязательных полей.

Плохо:

```ts
const price = body.price || 0
```

Хорошо:

```ts
if (body.price == null) {
  throw new ValidationError('price is required')
}
```

## Ошибки

- Ошибки не проглатываются молча.
- `catch {}` запрещён.
- Ожидаемые ошибки должны иметь понятный тип: `ValidationError`, `AuthenticationError`, `NotFoundError`.
- Сообщения ошибок должны быть конкретными.
- Внешнему пользователю не отдаём внутренние детали.
- Неожиданные ошибки логируются и превращаются в нейтральный ответ.

Плохо:

```ts
throw new Error('something went wrong')
```

Хорошо:

```ts
throw new ValidationError('Invalid booking status')
```

## Логирование

Логи должны быть структурированными.

Плохо:

```ts
console.error(err)
```

Хорошо:

```ts
logger.error({
  requestId,
  service: 'admin-service',
  action: 'resource.update',
  error: err.message,
})
```

Audit log обязателен для мутирующих административных и платёжных действий.

## Конфигурация

- Доступ к `process.env` централизован.
- Env валидируется при старте.
- Секреты не имеют небезопасных production fallback.
- Таймауты, лимиты и URL должны быть конфигурируемыми.
- Runtime config не читается внутри доменных модулей.

Допустимы безопасные local defaults только для локальной инфраструктуры, например localhost URL или локальный порт.

## Безопасность

Безопасность является частью архитектуры, а не отдельным этапом.

Обязательные практики:

- auth;
- authorization;
- rate limiting;
- replay protection;
- input validation;
- structured audit logs;
- least privilege;
- signed internal requests;
- encrypted secrets and tokens;
- request timeouts.

## Node.js

- HTTP, business logic и data access разделяются.
- Route handler должен быть тонким.
- Межсервисные запросы подписываются.
- Внешние вызовы имеют timeout.
- Тяжёлые операции не выполняются в request path.
- Используем `async/await`.
- Не используем скрытые global side effects.

## React

- Используем функциональные компоненты.
- Компонент решает одну задачу.
- Сложную логику выносим в hooks или services.
- UI-компоненты остаются декларативными.
- Важные формы делаем контролируемыми.
- Не создаём компоненты внутри render без сильной причины.
- Effects должны быть минимальными и понятными.

## Тестируемость

Любой код должен быть тестируемым.

Правила:

- бизнес-логику отделяем от транспорта;
- side effects изолируем;
- зависимости передаём явно;
- validation тестируем отдельно;
- критические сценарии и ошибки покрываем тестами.

Если код невозможно покрыть тестами, архитектура выбрана плохо.

## Форматирование

- Форматирование автоматическое.
- Отступ — 2 пробела.
- Длина строки — разумная.
- Политика по semicolons единая во всём проекте.
- Trailing commas — по настройке formatter.

## Имена файлов

- Обычные файлы — `kebab-case`.
- React-компоненты могут использовать `PascalCase`, если это принято в зоне проекта.
- Имя файла соответствует основному экспорту.

## Git-гигиена

- Один коммит — одна логическая задача.
- Не смешиваем рефакторинг и изменение поведения без необходимости.
- Переименования и переносы лучше делать отдельным коммитом.
- Перед merge код проходит форматирование, lint и typecheck.
- Code review обязателен для production-кода.

## Базовый стандарт репозитория

- TypeScript only.
- English names.
- Russian comments and JSDoc.
- No `any`.
- No silent fallbacks for required data.
- No giant files.
- No hidden side effects.
- Validation on every boundary.
- Structured errors.
- Structured logs.
- Explicit dependencies.
- Business logic outside UI and transport.
- Boring code over clever code.
