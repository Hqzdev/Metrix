# Техническое задание

## Проект

`Smart Booking System` — система управления бронированием переговорных комнат и рабочих мест в коворкинге.

## 1. Цель проекта

Создать систему, которая позволяет сотрудникам быстро находить свободные слоты, бронировать переговорные и рабочие места, а администраторам — видеть реальную загрузку ресурсов и получать аналитику по использованию помещений.

## 2. Пользователи системы

### Сотрудники

Используют систему для:

- поиска свободных переговорных;
- бронирования комнаты или рабочего места;
- переноса и отмены брони;
- получения напоминаний;
- быстрого бронирования через Telegram-бота.

### Администраторы

Используют систему для:

- управления помещениями и рабочими местами;
- просмотра статистики использования;
- контроля загрузки ресурсов;
- выгрузки отчётов;
- управления интеграциями и настройками.

## 3. Проблемы, которые решает система

- сотрудники не видят свободные слоты в одном окне;
- помещения используются неэффективно;
- бронирования трудно контролировать;
- нет прозрачной аналитики по занятости;
- события в корпоративных календарях не синхронизированы автоматически.

## 4. Функциональные требования

### 4.1 Управление бронированиями

Система должна позволять:

- просматривать список помещений и рабочих мест;
- видеть свободные и занятые слоты;
- создавать бронь на выбранное время;
- отменять бронь;
- переносить бронь на другой слот;
- получать подтверждение успешного бронирования;
- видеть список своих активных и завершённых броней.

### 4.2 Управление ресурсами

Система должна позволять администраторам:

- создавать переговорные комнаты и рабочие места;
- задавать вместимость, локацию и рабочие часы;
- указывать правила бронирования;
- временно делать ресурс недоступным;
- управлять статусами ресурсов.

### 4.3 Интеграция с календарями

Система должна поддерживать:

- авторизацию через Google;
- авторизацию через Microsoft;
- создание события в календаре после бронирования;
- обновление события при переносе брони;
- удаление события при отмене;
- повторные попытки синхронизации при временных ошибках внешнего API.

### 4.4 Telegram-бот

Telegram-бот должен поддерживать:

- просмотр доступных слотов;
- быстрое бронирование через кнопки;
- отмену и перенос доступных броней;
- напоминание за 15 минут до начала;
- быстрый доступ к ближайшим броням пользователя.

### 4.5 Аналитика

Система должна предоставлять администраторам:

- карту занятости по часам и дням;
- процент использования каждого помещения и рабочего места;
- пиковые часы бронирования;
- статистику отмен и переносов;
- фильтрацию по датам, ресурсам и локациям.

### 4.6 Отчёты

Система должна поддерживать:

- экспорт отчётов в PDF;
- выгрузку ключевых метрик за выбранный период;
- формирование отчётов по отдельным помещениям или локациям.

## 5. Нефункциональные требования

### 5.1 Производительность

- API должен отвечать менее чем за `500 мс` для 95% типовых запросов.
- Основные страницы должны загружаться менее чем за `3 секунды`.
- Обновление доступности должно происходить без заметной задержки для пользователя.

### 5.2 Безопасность

- все запросы должны идти только по `HTTPS`;
- внешняя авторизация должна быть реализована через `OAuth2`;
- сессии должны быть защищены;
- пароли, если локальная авторизация присутствует, должны храниться только в хешированном виде;
- доступ к административным функциям должен проверяться на backend;
- токены внешних интеграций должны храниться безопасно.

### 5.3 Доступность

- целевой uptime системы — `99%`;
- веб-интерфейс должен корректно работать на десктопе и мобильных устройствах;
- должны поддерживаться актуальные версии современных браузеров.

### 5.4 Поддерживаемость

- код должен быть разделён по доменным модулям;
- интеграции не должны протекать в бизнес-логику;
- архитектура должна позволять расширять систему без полного переписывания.

## 6. Роли и права

### Сотрудник

Может:

- просматривать доступные ресурсы;
- создавать, переносить и отменять свои брони;
- подключать личные календарные интеграции;
- пользоваться Telegram-ботом.

### Администратор

Может:

- управлять ресурсами;
- просматривать аналитику;
- экспортировать отчёты;
- управлять ограничениями и настройками системы;
- видеть общую загрузку компании.

## 8. Архитектурное решение

Для первой версии системы принимается **модульный монолит**, а не микросервисы.

Это решение выбрано потому, что:

- продукт находится на ранней стадии;
- основная логика тесно связана;
- на старте важнее скорость разработки и предсказуемость поддержки;
- очереди, realtime и интеграции можно добавить внутри одной системы без сервисного дробления.

Детальная схема зафиксирована в [ARCHITECTURE.md](./ARCHITECTURE.md).

## 10. Ключевые сценарии

### Бронирование

1. Пользователь выбирает ресурс.
2. Система показывает доступные слоты.
3. Пользователь создаёт бронь.
4. Бронь сохраняется в основной БД.
5. Во внешнем календаре создаётся событие.
6. Пользователь получает подтверждение.

### Отмена

1. Пользователь отменяет бронь.
2. Система снимает бронь в основной БД.
3. Связанное событие удаляется из календаря.
4. Доступность обновляется для других пользователей.

### Перенос

1. Пользователь выбирает новое время.
2. Система повторно валидирует доступность.
3. Бронь обновляется.
4. Событие в календаре изменяется.

## 11. Риски и технические ограничения

- внешние API Google и Microsoft могут отдавать временные ошибки;
- нужна защита от двойного бронирования при конкурентных запросах;
- аналитика не должна строиться тяжёлыми запросами в реальном времени;
- Telegram-бот не должен становиться отдельным источником истины;
- интеграции должны быть изолированы через адаптеры.
